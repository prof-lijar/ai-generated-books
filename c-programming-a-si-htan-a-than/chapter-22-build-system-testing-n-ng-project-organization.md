# Chapter 22: Build System, Testing နှင့် Project Organization

လက်တွေ့ကမ္ဘာရှိ C Programming project များသည် မည်သို့ စီမံခန့်ခွဲပုံနှင့် အကောင်းဆုံးအလေ့အထများအကြောင်း မိတ်ဆက်ပေးခြင်းဖြင့် စတင်လိုက်ရအောင်။ သင်စာသင်ခန်းများတွင် သင်ယူခဲ့သော C code တစ်ခုသည် စက်ကွန်ပျူတာ၏ အတွင်းပိုင်းတွင် မည်သို့ အလုပ်လုပ်ပုံကို နားလည်သော်လည်း၊ ထို code များကို စနစ်တကျ၊ မြန်ဆန်စွာ၊ မှန်ကန်စွာ အဖွဲ့လိုက် (သို့မဟုတ် ကိုယ်တိုင်) ဖန်တီး၊ စမ်းသပ်၊ ထိန်းသိမ်းနိုင်ရန် မည်သို့ စီစဉ်ရမည်ကို ဤအခန်းတွင် လေ့လာမည် ဖြစ်သည်။ ဤအခန်း၏ အဓိကရည်ရွယ်ချက်မှာ စာသင်ခန်းမှ လေ့လာခဲ့သော Syntax နှင့် Logic ကို Real-world application များအတွက် အသုံးချနိုင်ရန် Build System နှင့် Project Organization အခြေခံများကို တည်ဆောက်ပေးရန် ဖြစ်သည်။

## Real-world C Project ဖွဲ့စည်းပုံ (Project Structure)

လက်တွေ့တွင် ကျယ်ပြန့်သော project တစ်ခုကို စနစ်တကျ စီစဉ်ပုံနှင့် Directory Structure များသည် Code များ၏ ထိန်းသိမ်းမှုနှင့် ပရော်ဖက်ရှင်နယ်ဆန်မှုကို အဓိက ဆုံးဖြတ်ပေးသည်။ စနစ်တကျ စီစဉ်ထားသော ဖိုင်တွဲဖွဲ့စည်းပုံသည် အဖွဲ့လိုက် အလုပ်လုပ်ရာတွင် အလွန်အရေးကြီးသည်။ ထို့ကြောင့် ကျွန်ုပ်တို့သည် အောက်ပါ Directory Structure ကို အခြေခံအဖြစ် လက်ခံကျင့်သုံးကြရမည်။

အဆိုပါ စံဖွဲ့စည်းပုံတွင် အဓိကအားဖြင့် `src/`၊ `include/`၊ `lib/`၊ `test/` နှင့် `build/` ဟူသော Directory များ ပါဝင်သည်။ ဤဖွဲ့စည်းပုံတစ်ခုချင်းစီ၏ အခန်းကဏ္ဍများကို နားလည်ထားရန် လိုအပ်သည်။

`src/` (Source files): ဤနေရာတွင် ကျွန်ုပ်တို့၏ အရင်းအမြစ် (source) ဖိုင်များဖြစ်သော `.c` ဖိုင်များကို ထားရှိသည်။ ဤဖိုင်များသည် အမှန်တကယ် လုပ်ဆောင်ချက်များကို ကိုယ်စားပြုသည်။

`include/` (Header files): ဤနေရာတွင် `.h` ဖိုင်များ၊ သို့မဟုတ် Header files များကို ထားရှိသည်။ ဤဖိုင်များသည် အခြားသော Source files များက အသုံးပြုမည့် Function prototypes၊ structure definitions နှင့် macro များကို ထည့်သွင်းထားသည်။ ဤအရာက Code များအကြား ဆက်သွယ်မှုကို အလွယ်တကူ ဖြစ်စေသည်။

`lib/` (Library files): ဤနေရာသည် ကျွန်ုပ်တို့ဖန်တီးထားသော Library များ၊ Static Library များ (.a) နှင့် Shared Library များ (.so/.dylib) များကို ထားရှိသည်။ ဤနေရာတွင် Code များအား ပြန်လည်အသုံးပြုနိုင်ရန်အတွက် စုစည်းထားသည်။

`test/` (Testing files): Unit testing များအတွက် လိုအပ်သော Test code များ၊ Test harness များ နှင့် Test case များကို ထားရှိသည်။ ဤအပိုင်းသည် Code များ၏ မှန်ကန်မှုကို စစ်ဆေးရန် အထောက်အကူပြုသည်။

`build/` (Build artifacts): ဤနေရာသည် Compile လုပ်ပြီးသော အရာများကို ထားရှိသည်။ ဥပမာအားဖြင့် Object files (.o) များ၊ Static Library များ (.a) နှင့် နောက်ဆုံးထုတ်ပေးသော Executable ဖိုင်များကို ထားရှိသည်။ ဤဖိုင်များကို ထိန်းသိမ်းထားခြင်းဖြင့် နောက်တစ်ကြိမ် Compile လုပ်သည့်အခါ အချိန်ကုန်သက်သာစေသည်။

ဤဖွဲ့စည်းပုံသည် Code များ၏ အရင်းအမြစ်များ၊ ကြေညာချက်များ၊ စမ်းသပ်မှုများနှင့် အထွက်ရလဒ်များကို ခွဲခြားသိမြင်နိုင်စေပြီး ပရောဂျက်တစ်ခုလုံးကို စီမံခန့်ခွဲရန် အထောက်အကူပြုသည်။ ဤဖွဲ့စည်းပုံကို စနစ်တကျ ထားရှိနိုင်ရန်အတွက် အလိုအလျောက် စီမံခန့်ခွဲပေးမည့် ကိရိယာတစ်ခု လိုအပ်သည်။ ထို့ကြောင့် နောက်အပိုင်းတွင် Build System အကြောင်းကို မိတ်ဆက်ပေးမည်။

## Makefile နှင့် Compilation Process (The Heart of Build System)

ဤဖွဲ့စည်းပုံကို စနစ်တကျ ထားရှိနိုင်ရန်နှင့် C code များကို အလိုအလျောက် Compile လုပ်ရန်နှင့် Link လုပ်ရန်အတွက် အကောင်းဆုံး နည်းလမ်းမှာ `Makefile` ကို အသုံးပြုခြင်းပင် ဖြစ်သည်။ `Makefile` သည် Compilation လုပ်ငန်းစဉ်တစ်ခုလုံးကို အလိုအလျောက် စီမံခန့်ခွဲပေးသော စာသားဖိုင်တစ်ခု ဖြစ်သည်။

### Makefile အခြေခံများ

`Makefile` တွင် အခြေခံအားဖြင့် အချက်လေးမျိုးကို နားလည်ရန် လိုအပ်သည်။ ၎င်းတို့မှာ Target၊ Dependency၊ Recipe၊ နှင့် Variable များ ဖြစ်သည်။

Target (ပစ်မှတ်): ၎င်းသည် ကျွန်ုပ်တို့ လိုချင်သော အရာ (ဥပမာ- Executable ဖိုင်) ဖြစ်သည်။
Dependency (မှီခိုမှု): ၎င်းသည် Target ထုတ်လုပ်ရန် လိုအပ်သော ဖိုင်များ (ဥပမာ- Source file များ သို့မဟုတ် Header file များ) ဖြစ်သည်။
Recipe (လုပ်ဆောင်ပုံ): ၎င်းသည် Target ကို ရရှိရန်အတွက် လုပ်ဆောင်ရမည့် အဆင့်ဆင့်သော Command များ (ဥပမာ- `gcc` command) ဖြစ်သည်။
Variable (အပြောင်းအလဲ): ၎င်းသည် အသုံးပြုမည့် တန်ဖိုးများ (ဥပမာ- Compiler flags, file names) ကို သတ်မှတ်ထားခြင်း ဖြစ်သည်။

Dependency Management (မှီခိုမှု စီမံခန့်ခွဲမှု): Dependency Management သည် အလွန်အရေးကြီးသည်။ ဖိုင်တစ်ခု ပြောင်းလဲသွားပါက မည်သည့်ဖိုင်များကို ပြန်လည် Compile လုပ်ရမည်ကို စနစ်တကျ သိရှိခြင်းသည် အချိန်ကုန်သက်သာစေသည်။ ဥပမာအားဖြင့်၊ Source file တစ်ခု ပြောင်းသွားပါက ထို Source file ကို ပြန် Compile လုပ်ပြီး၊ ထို Source file မှ မှီခိုသော Object file (.o) များကို ပြန် Compile လုပ်ရန် လိုအပ်သည်။

Compilation Flow (Compile လုပ်ခြင်း၏ စီးဆင်းမှု): C project များတွင် Compilation သည် အဆင့်ဆင့် ဖြစ်သည်။ ပထမအဆင့်မှာ Separate Compilation (ခွဲ၍ Compile လုပ်ခြင်း) ဖြစ်သည်။ ဤအဆင့်တွင် `.c` ဖိုင်များကို `.o` (Object file) များအဖြစ် Compile လုပ်ပြီး၊ နောက်ဆုံးအဆင့်တွင် Linking (ဆက်စပ်ခြင်း) ဖြစ်သည်။ Linking သည် Object files နှင့် Library များကို ပေါင်းစပ်၍ နောက်ဆုံး Executable ဖိုင်ကို ထုတ်ပေးသည်။

Pattern Rule (ပုံစံ စည်းမျဉ်း): Pattern Rule များသည် `Makefile` ကို ပိုမိုတိုတောင်းပြီး ထိရောက်စွာ ရေးသားနိုင်ရန် ကူညီပေးသည်။ ဥပမာအားဖြင့်၊ `%.c` သို့မဟုတ် `%.o` ကဲ့သို့သော Pattern Rule များ အသုံးပြု၍ အချိန်ကုန်သက်သာစွာ Compile လုပ်နည်းများကို ဖော်ပြနိုင်သည်။ ဤနည်းလမ်းက တစ်ခုတည်းသော Source file ကို Compile လုပ်မည့်အစား၊ အခြား Source file အများအပြားကို တစ်ပြိုင်နက်တည်း စီမံခန့်ခွဲနိုင်စေသည်။

Variable နှင့် Function (Variable နှင့် လုပ်ဆောင်ချက်များ): Makefile အတွင်းမှ Variable များ (ဥပမာ- `CC`, `CFLAGS`) နှင့် Function များ အသုံးပြုပုံကိုလည်း လေ့လာရမည်။ ဤနည်းလမ်းက ပရောဂျက်တစ်ခုလုံးအတွက် တစ်သမတ်တည်းရှိသော Compile လုပ်မှုများကို အာမခံပေးသည်။

## Libraries: Static နှင့် Shared (Static and Shared Libraries)

Code များအား ပြန်လည်အသုံးပြုနိုင်ရန်အတွက် Library များသည် မရှိမဖြစ်လိုအပ်သည်။ Library များသည် Code အစုအဝေးများကို စုစည်းထားခြင်း ဖြစ်သည်။ ဤ Library များအား Static နှင့် Shared အမျိုးအစားများအဖြစ် ခွဲခြားနိုင်သည်။

Static Libraries (.a): Static Library များသည် Code များအား Compile လုပ်ပြီးသော Object files များကို စုစည်းထားခြင်း ဖြစ်သည်။ ၎င်းတို့ကို ဖန်တီးရန်အတွက် `ar` နှင့် `arrc` ကဲ့သို့သော Command များကို အသုံးပြုသည်။ Static Library များသည် Compile လုပ်စဉ်အတွင်း Data များအားလုံးကို Code ထဲသို့ ထည့်သွင်းထားသည်။ Memory တွင် Data များ မည်သို့ သိမ်းဆည်းပုံကို Static Library များက Shared Library များနှင့် နှိုင်းယှဉ်ချက်ဖြင့် နားလည်ထားရန် အရေးကြီးသည်။

Shared Libraries (.so / .dylib): Shared Library များသည် Code များအား Compile လုပ်ပြီးသော Object files များကို မထည့်ဘဲ၊ Runtime တွင် အသုံးပြုရန်အတွက် ဖန်တီးထားသည်။ ၎င်းတို့ကို Dynamic Linking ဖြင့် အသုံးပြုသည်။ Shared Library များသည် Runtime တွင် အသုံးပြုရန်အတွက် လိုအပ်သော Code များအား Linker မှ တောင်းဆိုမှသာ ထည့်သွင်းပေးသည်။

Library Linking (Library ဆက်စပ်ခြင်း): Compiler (Linker) သည် Library များနှင့် မည်သို့ အပြန်အလှန် ဆက်သွယ်ပုံကို နားလည်ခြင်းသည် အရေးကြီးသည်။ Linker သည် Object files များရှိ Code များ၏ Address များကို Library များရှိ Function များ၏ Address များနှင့် တိုက်ဆိုင်စစ်ဆေးပြီး မှန်ကန်စွာ ဆက်စပ်ပေးသည်။

Practical Example (လက်တွေ့ဥပမာ): ပရောဂျက်တစ်ခုတွင် Static Library တစ်ခုကို ဖန်တီး၍ အသုံးပြုပုံကို လက်တွေ့ပြသခြင်း။ လက်တွေ့တွင် ပရောဂျက်တစ်ခုကို အစမှအဆုံး ဖန်တီးရာတွင် Static Library များသည် Compile လုပ်ပြီးနောက် ထွက်လာသော Code များအားလုံးကို ထည့်သွင်းရန် လွယ်ကူစေသည်။

## Unit Testing နှင့် Code Style (Quality Assurance)

Code များ၏ မှန်ကန်မှုကို စစ်ဆေးခြင်းနှင့် ပရော်ဖက်ရှင်နယ်ဆန်သော Code များ ရေးသားခြင်းဆိုင်ရာ အခြေခံများကို လေ့လာရမည်။

Unit Testing Framework (ယူနစ် စမ်းသပ်မှု မူဘောင်): Code များ၏ သေးငယ်သော အပိုင်းအစများကို စစ်ဆေးရန် Unit Testing သည် မရှိမဖြစ်လိုအပ်သည်။ ကျွန်ုပ်တို့သည် Assert-based Test Framework တစ်ခုကို ရိုးရှင်းစွာ ရေးသားခြင်း (ဥပမာ: `assert()` ကို အခြေခံ၍) ကို လေ့လာမည်။ ဤနည်းလမ်းက Test Case များ ရေးသားပုံနှင့် Test လုပ်ဆောင်ပုံကို ပရော်ဖက်ရှင်နယ်ဆန်စွာ သင်ကြားပေးသည်။

Testing Philosophy (စမ်းသပ်မှု အတွေးအခေါ်): Unit Test များ၏ အရေးပါပုံနှင့် Test Case များ ရေးသားပုံကို နားလည်ရန် လိုအပ်သည်။ Test Case များသည် Code ၏ ဖြစ်နိုင်ခြေရှိသော အခြေအနေများ (edge cases များအပါအဝင်) ကို စမ်းသပ်ရန် အထောက်အကူပြုသည်။

Code Style & Naming Convention (Code ပုံစံနှင့် အမည်ပေးခြင်း): ပရော်ဖက်ရှင်နယ်ဆန်သော Code များ ရေးသားရန်အတွက် Code Style နှင့် Naming Convention များသည် အလွန်အရေးကြီးသည်။

Variable, Function, File နာမည်ပေးခြင်းဆိုင်ရာ စည်းမျဉ်းများ (Naming Convention): Variable များ၊ Function များ၊ နှင့် File များ အမည်ပေးရာတွင် သတ်မှတ်ထားသော စည်းမျဉ်းများကို လိုက်နာရမည်။ ဥပမာအားဖြင့်၊ အင်္ဂလိပ်စာလုံးများဖြင့် စတင်ပြီး အဓိပ္ပာယ်ရှိစေရန် စီစဉ်ရမည်။

Indentation နှင့် Code Organization များ၏ အရေးပါပုံ: Code များ၏ Indentation (အလင်းတန်း သို့မဟုတ် အကွာအဝေး) နှင့် Code Organization များသည် Code ကို ဖတ်ရှုရလွယ်ကူစေသည်။ မှန်ကန်သော Indentation သည် Code ၏ Logic ကို မှန်ကန်စွာ ဖော်ပြပေးသည်။

Documentation (စာတမ်းတင်ခြင်း): Code Comment များသည် Function များ၊ Structure များ၊ နှင့် Logic များကို ရှင်းပြရန်အတွက် မရှိမဖြစ်လိုအပ်သည်။ Code Comment များသည် ကိုယ်တိုင်နှင့် အခြားသူများအတွက် Code ၏ ရည်ရွယ်ချက်ကို ရှင်းလင်းစေသည်။ ထို့အပြင် README ဖိုင်သည် ပရောဂျက်၏ အကျဉ်းချုပ်၊ ထည့်သွင်းပုံနှင့် အသုံးပြုပုံများကို ဖော်ပြရန် အရေးကြီးသည်။

## Version Control နှင့် လက်တွေ့အသုံးချမှု (Version Control and Practical Implementation)

Code များ၏ မှတ်တမ်းများကို ထိန်းသိမ်းခြင်းနှင့် အဖွဲ့လိုက် လုပ်ဆောင်ခြင်းအတွက် Version Control နည်းပညာကို မိတ်ဆက်ပေးမည်။ ထိုနည်းလမ်းမှာ Git ဖြစ်သည်။

### Git အခြေခံများ

Git ကို စတင်အသုံးပြုရန်အတွက် အခြေခံအချက်များကို လေ့လာရမည်။

Git ကို စတင်ခြင်း (`init`): ပရောဂျက်တစ်ခုအတွက် Git Repository ကို စတင်တည်ထောင်ခြင်း။
ဖိုင်များ ထည့်သွင်းခြင်း (`add`): ပြောင်းလဲထားသော ဖိုင်များကို Git စနစ်ထဲသို့ ထည့်သွင်းခြင်း။
မှတ်တမ်းတင်ခြင်း (`commit`): ပြောင်းလဲမှုများကို မှတ်တမ်းတင်ခြင်း။
History Management (သမိုင်း စီမံခန့်ခွဲမှု): မှတ်တမ်းများကို နှိုင်းယှဉ်ခြင်း (`diff`) နှင့် မှတ်တမ်းများ၏ သမိုင်းကို ကြည့်ရှုခြင်း (`log`) တို့သည် အရေးကြီးသည်။
Branching (ခွဲခြားခြင်း): Branch များ ဖန်တီးခြင်းဖြင့် အလုပ်များကို သီးခြားစီ စီမံခန့်ခွဲခြင်း။ ဤနည်းလမ်းက အဓိက Development လုပ်နေသော အပိုင်းကို အခြားအပိုင်းများနှင့် မရောထွေးစေဘဲ အလုပ်လုပ်နိုင်စေသည်။

Practical Project Implementation (လက်တွေ့ ပရောဂျက် အကောင်အထည်ဖော်ခြင်း): အထက်ပါ အချက်များအားလုံးကို ပေါင်းစပ်၍ Multiple Source File များ၊ Header File များ၊ Test File များ နှင့် Makefile တစ်ခုပါဝင်သော လက်တွေ့ Project တစ်ခုကို အစမှအဆုံး ဖန်တီးခြင်းကို လက်တွေ့လုပ်ဆောင်ကြမည်။ ဤအဆင့်တွင် ကျွန်ုပ်တို့သည် အထက်တွင် လေ့လာခဲ့သော Build System၊ Testing၊ Code Style နှင့် Version Control တို့ကို တစ်ပြိုင်နက်တည်း အသုံးချသွားမည် ဖြစ်သည်။

## နိဂုံးချုပ် (Conclusion/Summary)

ဤအခန်းတွင် ကျွန်ုပ်တို့သည် C Programming ကို ပရော်ဖက်ရှင်နယ် အဆင့်သို့ မြှင့်တင်ရန် လိုအပ်သော အခြေခံအုတ်မြစ်များကို တည်ဆောက်ခဲ့ကြသည်။ Build System များ (Makefile) သည် Code များကို အလိုအလျောက် စီမံခန့်ခွဲရန်၊ Libraries များ (Static/Shared) သည် Code များအား ပြန်လည်အသုံးပြုရန်၊ Unit Testing များသည် Code များ၏ မှန်ကန်မှုကို အာမခံရန်၊ Code Style နှင့် Documentation များသည် Code များကို ဖတ်ရှုရလွယ်ကူစေရန်၊ နှင့် Version Control (Git) သည် ပရောဂျက်များ၏ မှတ်တမ်းများကို ထိန်းသိမ်းရန် အရေးကြီးပါသည်။

ဤအချက်များအားလုံးသည် C Programming ကို မည်သို့ စီမံခန့်ခွဲရမည်၊ စမ်းသပ်ရမည်၊ ထိန်းသိမ်းရမည်ကို အကျုံးဝင်သည်။ လက်တွေ့တွင် ကျယ်ပြန့်သော C Programming project များသည် ဤအခြေခံများပေါ်တွင် တည်ဆောက်ထားပြီး၊ ဤအခြေခံများကို ကျွမ်းကျင်စွာ အသုံးချနိုင်သူများသည် ပရော်ဖက်ရှင်နယ် ဆန်သော Software များ ဖန်တီးနိုင်မည် ဖြစ်သည်။ Build System, Testing, Code Style, နှင့် Version Control တို့သည် C Programming ကို ပရော်ဖက်ရှင်နယ် အဆင့်သို့ မြှင့်တင်ရန် အဓိကသော့ချက်များ ဖြစ်ကြောင်း ဤအခန်းမှ အနှစ်ချုပ် ဖော်ပြလိုက်ရသည်။
