# Chapter 20: Error Handling နှင့် Defensive Programming

## ၁။ အဖွင့်အစနှင့် မိတ်ဆက် (Introduction and Hook)

C Programming ကို သင်ယူနေသူတိုင်းသည် ပရိုဂရမ်တစ်ခုကို ရေးသားသည့်အခါ အမှားအယွင်းများနှင့် ရင်ဆိုင်ရမည်ကို မလွဲမရှောင်သာ သိရှိရမည် ဖြစ်သည်။ C Language သည် ကွန်ပျူတာ၏ အခြေခံနှင့် အနီးစပ်ဆုံး ဘာသာစကားဖြစ်သော်လည်း၊ ၎င်းတွင် Exception Handling (အမှားကို ကိုင်တွယ်ဖြေရှင်းခြင်း) ကဲ့သို့သော ပရိုဂရမ်မင်းဘာသာစကားများတွင် အမှားကို အလိုအလျောက် ဖြေရှင်းပေးသည့် စနစ် (Exception Handling) မရှိပေ။ ထို့ကြောင့် ကျွန်ုပ်တို့သည် Error Handling နှင့် Defensive Programming (ကာကွယ်ဆင်ခြင်မှု) ဆိုသော နည်းစနစ်များကို အသုံးပြု၍ ပရိုဂရမ်များ၏ ခိုင်မာမှု (Reliability) နှင့် လုံခြုံရေး (Security) ကို ကိုယ်တိုင် ထိန်းချုပ်ရမည် ဖြစ်သည်။

လက်တွေ့ဘဝတွင် ကျွန်ုပ်တို့၏ လုပ်ဆောင်မှုများသည် အမြဲတမ်း မှန်ကန်မည်မဟုတ်ဘဲ၊ ထင်မှတ်မထားသော အခြေအနေများ (ဥပမာ- အသုံးပြုသူမှ မှားယွင်းသော အချက်အလက်များ ထည့်သွင်းခြင်း၊ စနစ်မှ ထွက်ပေါ်လာသော အမှားများ) ကြောင့် ပရိုဂရမ်များသည် ရပ်တန့်သွားခြင်း သို့မဟုတ် မှားယွင်းသော ရလဒ်များကို ထုတ်ပေးခြင်းများ ဖြစ်တတ်သည်။ ထိုအမှားများကို ရှောင်ရှားရန်နှင့် ဖြစ်ပေါ်လာပါက ထိန်းချုပ်နိုင်ရန်အတွက် Error Handling နှင့် Defensive Programming များသည် မရှိမဖြစ် လိုအပ်လာသည်။ ဤအခန်းတွင် ကျွန်ုပ်တို့သည် Return Value များ၊ System Error Codes များ၊ နှင့် ပရိုဂရမ်ကို ပိုမိုခိုင်မာစေမည့် ရှေ့တန်းနည်းစနစ်များ (Defensive Programming) များကို အသေးစိတ် လေ့လာသွားကြမည် ဖြစ်သည်။

ကျွန်ုပ်တို့၏ ရည်မှန်းချက်မှာ C Programming တွင် အမှားများကို လက်ခံနိုင်သော ပုံစံဖြင့် ကိုင်တွယ်ဖြေရှင်းတတ်ရန်၊ ထို့အပြင် ပရိုဂရမ်များအား အလွန်အမင်း ခိုင်မာစေရန်နှင့် မမျှော်လင့်သော အခြေအနေများတွင်ပင် မှန်ကန်စွာ ဆက်လက်အလုပ်လုပ်နိုင်စေရန် နည်းစနစ်များကို ကျွမ်းကျင်စွာ အသုံးချတတ်ရန် ဖြစ်သည်။

## ၂။ C တွင် Error Handling ၏ အခြေခံ (Fundamentals of Error Handling in C)

C Language တွင် Exception Handling ကဲ့သို့သော စနစ်မရှိသောကြောင့် Error Handling ကို ကျွန်ုပ်တို့ကိုယ်တိုင် တည်ဆောက်ရမည် ဖြစ်သည်။ ဤအခြေခံများကို နားလည်ခြင်းဖြင့် ကျွန်ုပ်တို့သည် ပရိုဂရမ်များ၏ အမှားများကို သိရှိပြီး စနစ်တကျ ကိုင်တွယ်နိုင်မည် ဖြစ်သည်။

### Return Value များဖြင့် Error ကို ဖော်ပြခြင်း

Function တစ်ခုသည် အလုပ်လုပ်ပြီးကြောင်း သို့မဟုတ် အမှားဖြစ်ကြောင်းကို ပြင်ပသို့ အကြောင်းကြားရန် Return Value များသည် အဓိကကျသည်။ Function များသည် အလုပ်ပြီးမြောက်ပါက အထူးတန်ဖိုး (ဥပမာ- 0) ကို ပြန်ပေးပြီး အမှားဖြစ်ပေါ်ပါက အမှားကို ဖော်ပြရန်အတွက် အထူးတန်ဖိုးများကို အသုံးပြုရသည်။

Error ကို ဖော်ပြရန်အတွက် အထူးတန်ဖိုးများကို အသုံးပြုလေ့ရှိသည်။ ဥပမာအားဖြင့်၊ အောင်မြင်ကြောင်း ဖော်ပြရန်အတွက် `0` ကို အသုံးပြုနိုင်ပြီး၊ အမှားဖြစ်ပေါ်ကြောင်း ဖော်ပြရန်အတွက် `1` သို့မဟုတ် အနုတ်လက္ခဏာ (ဥပမာ- `-1`) ကို အသုံးပြုနိုင်သည်။ ထို့အပြင် Pointer များကို အသုံးပြုသောအခါ အမှားမရှိကြောင်း ဖော်ပြရန်အတွက် `NULL` (သို့မဟုတ် `NULL` pointer) ကို အသုံးပြုလေ့ရှိသည်။ ဤနည်းလမ်းသည် Function တစ်ခုမှ ပြန်ပေးသော တန်ဖိုးကို စစ်ဆေးခြင်းဖြင့် အမှားကို ဖော်ထုတ်ရန် အခြေခံကျသည်။

### System Error Codes နှင့် errno

ပရိုဂရမ်များ အလုပ်လုပ်နေစဉ်အတွင်း Operating System (OS) မှ ဖြစ်ပေါ်လာသော အမှားများကို သိရှိရန်အတွက် System Error Codes များသည် အလွန်အရေးကြီးသည်။ ဤအမှားများကို သိရှိနိုင်ရန်အတွက် C Standard Library တွင် ပါဝင်သော `errno` (Error Number) ဆိုသော Variable ကို အသုံးပြုသည်။ `errno` သည် လက်ရှိဖြစ်ပေါ်နေသော အမှားကို ဖော်ပြသည့် ဂဏန်းတန်ဖိုးကို သိမ်းဆည်းထားသည်။

OS မှ အမှားများ ဖြစ်ပေါ်လာသည့်အခါ၊ ကျွန်ုပ်တို့သည် `perror()` နှင့် `strerror()` functions များကို အသုံးပြုနိုင်သည်။ `perror(const char *s)` function သည် `errno` တွင် သိမ်းဆည်းထားသော အမှားကို ဖော်ပြသည့် စာသားကို `stderr` (Standard Error) တွင် ထုတ်ပေးသည်။ `strerror(int errnum)` function သည် `errno` ၏ ဂဏန်းတန်ဖိုးကို ဖတ်၍ အဆိုပါ အမှားနှင့် သက်ဆိုင်သော ဖော်ပြချက်စာသားကို ပြန်ပေးသည်။ ဤ function များသည် ပရိုဂရမ်၏ အမှားများကို စနစ်တကျ မှတ်တမ်းတင်ရန်နှင့် အသုံးပြုသူထံသို့ ရှင်းလင်းစွာ ပြသရန် အထောက်အကူပြုသည်။

### Error Handling အတွက် အထူးနည်းလမ်းများ (Advanced Techniques)

အမှားများကို အဆင့်ဆင့် ကိုင်တွယ်ဖြေရှင်းရန်အတွက် ပိုမိုရှုပ်ထွေးသော နည်းလမ်းများလည်း ရှိသည်။ ၎င်းတို့ထဲမှ တစ်ခုမှာ `setjmp` နှင့် `longjmp` Approach ဖြစ်သည်။ ဤနည်းလမ်းသည် Function Call Stack (Function များ၏ အစီအစဉ်) ကို အသုံးပြု၍ အမှားတစ်ခုမှ တစ်ဖက်သို့ ခုန်သွားစေမည့် နည်းလမ်းဖြစ်သည်။ `setjmp()` function သည် အမှားမဖြစ်မီ အခြေအနေတစ်ခုကို သိမ်းဆည်းထားပြီး၊ `longjmp()` function ကို အသုံးပြု၍ အမှားဖြစ်ပေါ်သည့် နေရာမှ အစောပိုင်းအခြေအနေသို့ ခုန်သွားစေသည်။ ဤနည်းလမ်းသည် Exception Handling နှင့် ဆက်စပ်မှုရှိသော အခြေခံသဘောတရားတစ်ခုဖြစ်ပြီး၊ Function များကြားတွင် အမှားကို ပိုမိုထိရောက်စွာ စီမံခန့်ခွဲနိုင်စေသည်။

## ၃။ Development Time တွင် အမှားစစ်ဆေးခြင်း (Checking Assumptions at Development Time)

ပရိုဂရမ်တစ်ခုကို ရေးသားသည့်အချိန် (Development Time) တွင် အမှားများကို ရှာဖွေပြီး ပြင်ဆင်ခြင်းသည် Runtime (ပရိုဂရမ်အလုပ်လုပ်နေစဉ်) တွင် အမှားများကို ဖြေရှင်းခြင်းထက် ပိုမိုလွယ်ကူသည်။ ထို့ကြောင့် Defensive Programming (ကာကွယ်ဆင်ခြင်မှု) နည်းစနစ်များကို အသုံးပြုရန် လိုအပ်သည်။

### `assert()` Function ၏ အသုံးပြုပုံ

`assert()` function သည် Development Time တွင် ကျွန်ုပ်တို့၏ Assumption များကို စစ်ဆေးရန် အလွန်အသုံးဝင်သည်။ ဤ function သည် တစ်ခုတည်းသော ရည်ရွယ်ချက်မှာ သတ်မှတ်ထားသော အခြေအနေ (condition) မှန်မမှန်ကို စစ်ဆေးရန် ဖြစ်သည်။ ဥပမာအားဖြင့်၊ တစ်ခုခုသည် အမြဲတမ်း မှန်ကန်ရမည်ဟု ကျွန်ုပ်တို့ ယူဆထားသည့် အချက်များကို စစ်ဆေးရန်အတွက် `assert(condition);` ကို အသုံးပြုသည်။

`assert()` သည် Runtime တွင် အမှားကို မည်သို့ ထုတ်ပေးပုံကို ရှင်းပြရမည်ဆိုလျှင်၊ `assert()` သည် စစ်ဆေးမှု မအောင်မြင်ပါက၊ ပရိုဂရမ်ကို **segmentation fault** (သို့မဟုတ်) သတ်မှတ်ထားသော အမှားအကြောင်းအရာကို ထုတ်ပေးပြီး အလုပ်လုပ်ခြင်းကို ရပ်တန့်စေသည်။ ၎င်းသည် Debugging လုပ်သည့်အချိန်တွင် အမှား၏ အရင်းအမြစ်ကို အလွန်လျင်မြန်စွာ ဖော်ထုတ်ပေးသည်။

### Input Validation ၏ အရေးပါမှု

User Input များသည် ကျွန်ုပ်တို့ ထင်မှတ်မထားသော ပုံစံများဖြင့် ရရှိလာနိုင်သောကြောင့် Input Validation (အချက်အလက် စစ်ဆေးခြင်း) သည် အလွန်အရေးကြီးသည်။ အသုံးပြုသူထံမှ ရရှိသော Input များသည် မှန်ကန်မှုရှိမရှိ၊ လိုအပ်သော ပုံစံနှင့် အရှည်ရှိမရှိ စသည်တို့ကို စစ်ဆေးခြင်းဖြင့် ပရိုဂရမ်၏ ခိုင်မာမှုကို သိသိသာသာ တိုးတက်စေသည်။ Input ကို စစ်ဆေးခြင်းသည် ပရိုဂရမ်သည် မမျှော်လင့်သော အခြေအနေများတွင် ပျက်စီးသွားခြင်းမှ ကာကွယ်ပေးသည်။

### Buffer Overflow ကို ကာကွယ်ခြင်း

C Programming တွင် Array များ (အစီအစဉ်များ) နှင့် Pointer များအသုံးပြုသောအခါ `buffer overflow` (အလွန်ကြီးသော အချက်အလက်ကို သတ်မှတ်ထားသော နေရာထက် ပိုမိုရေးထည့်ခြင်း) ဆိုသော အန္တရာယ်ကြီးတစ်ခု ရှိသည်။ ဤအမှားသည် ပရိုဂရမ်၏ မှားယွင်းသော လုပ်ဆောင်ချက်များ၊ လုံခြုံရေး ချို့ယွင်းမှုများ၊ သို့မဟုတ် စနစ်တစ်ခုလုံးကို ထိခိုက်စေနိုင်သော အန္တရာယ်ရှိသည်။ Buffer Overflow ဖြစ်ပေါ်ပုံမှာ Pointer များက သတ်မှတ်ထားသော Memory နေရာကို ကျော်လွန်၍ ရေးထည့်ခြင်းကြောင့် ဖြစ်ပေါ်သည်။ ဤအန္တရာယ်ကို ရှောင်ရှားရန်အတွက် Array များ၏ အစအဆုံး (boundary) ကို အမြဲတမ်း စစ်ဆေးပြီး၊ String များ ကိုင်တွယ်ရာတွင်လည်း အထူးဂရုစိုက်ရန် လိုအပ်သည်။

## ၄။ လုံခြုံရေးနှင့် ဘေးကင်းသော String စီမံခန့်ခွဲမှု (Safe String Handling and Security)

String များကို ကိုင်တွယ်ရာတွင် Buffer Overflow နှင့် အခြားသော လုံခြုံရေး ချို့ယွင်းမှုများ ဖြစ်ပေါ်နိုင်သောကြောင့် အထူးဂရုစိုက်ရမည်။

### အန္တရာယ်ရှိသော String Functions များမှ ရှောင်ရှားခြင်း

C Language တွင် String များကို ကိုင်တွယ်ရာ၌ အန္တရာယ်များသော function များရှိသည်။ အထူးသဖြင့် `gets()` function သည် အလွန်အန္တရာယ်များသည်။ `gets()` function သည် Input ကို ဖတ်ယူရာတွင် အသုံးပြုသူမှ ထည့်သွင်းသော အချက်အလက်၏ အရှည်ကို စစ်ဆေးခြင်းမရှိဘဲ Buffer ထဲသို့ တိုက်ရိုက်ရေးထည့်သည်။ ဤနည်းလမ်းကြောင့် Buffer Overflow ဖြစ်ပေါ်နိုင်ပြီး၊ ပရိုဂရမ်၏ Memory ကို ပျက်စီးစေနိုင်သည်။ ထို့ကြောင့် `gets()` ကို အစားထိုးရန်အတွက် ပိုမိုလုံခြုံသော `fgets()` function ကို အသုံးပြုသင့်သည်။ `fgets()` function သည် ဖတ်ယူမည့် Buffer ၏ အမြင့်ဆုံး အရှည်ကို သတ်မှတ်ပေးနိုင်သောကြောင့် Buffer Overflow ကို ကာကွယ်ပေးသည်။

### String Manipulation Functions များ

String များကို ကူးယူခြင်း (Copying) နှင့် ပုံစံသွင်းခြင်း (Formatting) အတွက်လည်း ဘေးကင်းသော နည်းလမ်းများကို အသုံးပြုရမည်။

`strncpy()` function သည် String တစ်ခုကို အခြားနေရာသို့ ကူးယူရာတွင် အသုံးပြုသည်။ ၎င်းသည် ကူးယူမည့် အပိုင်းအစ အရေအတွက်ကို သတ်မှတ်ပေးနိုင်သောကြောင့် Buffer Overflow ကို အနည်းငယ် လျှော့ချပေးသည်။ သို့သော် ၎င်းသည် String ကို အဆုံးသတ် (null-terminated) မလုပ်ပေးနိုင်သောကြောင့် သတိထားရန် လိုသည်။ ထို့ကြောင့် `snprintf()` function ကို အသုံးပြုခြင်းသည် String ကို ပုံစံသွင်းရာတွင် အလွန်ဘေးကင်းသည်။ `snprintf()` သည် ထည့်သွင်းမည့် စာသား၏ အရှည်ကို ထိန်းချုပ်နိုင်ပြီး၊ Buffer Overflow ဖြစ်နိုင်ခြေကို သိသိသာသာ လျှော့ချပေးသည်။ String Length နှင့် Pointer များကြား ကွာခြားချက်ကို နားလည်ထားရန် လိုအပ်သည်၊ Pointer များသည် Memory ၏ လိပ်စာကို ကိုယ်စားပြုပြီး String များသည် အဲဒီလိပ်စာများပေါ်ရှိ စာ Zeichen များ၏ စီးဆင်းမှုကို ကိုယ်စားပြုသည်။

### အခြားသော Security Vulnerabilities များ

String နှင့် Integer များကို ကိုင်တွယ်ရာတွင် အခြားသော လုံခြုံရေး ချို့ယွင်းမှုများလည်း ရှိသည်။ `format string attack` ဆိုသည်မှာ `printf()` ကဲ့သို့သော function များတွင် ထည့်သွင်းလိုက်သော format string ၏ အကြောင်းအရာကို ထိန်းချုပ်နိုင်သော အချက်အလက်များကို မှားယွင်းစွာ ထည့်သွင်းခြင်းကြောင့် ဖြစ်ပေါ်လာသော အန္တရာယ်ဖြစ်သည်။ ထို့အပြင် `integer overflow` (ဂဏန်းများ၏ အလွန်ကြီးမားသော ရလဒ်များ) သည်လည်း အရေးကြီးသည်။ `integer overflow` ဖြစ်ပေါ်ပုံမှာ အလွန်ကြီးမားသော ဂဏန်းများကို ပေါင်းလဒ်လုပ်သောအခါ၊ ထိုရလဒ်သည် သတ်မှတ်ထားသော `int` (သို့မဟုတ် `short`) data type ၏ လက်ခံနိုင်သော အပိုင်းအစထက် ကျော်လွန်သွားပြီး အလွန်သေးငယ်သော သို့မဟုတ် အနုတ်လက္ခဏာဖြင့် ပြောင်းလဲသွားခြင်းကြောင့် ဖြစ်ပေါ်သည်။ ဤအမှားကို ကာကွယ်ရန်အတွက် `unsigned` data type များကို အသုံးပြုခြင်းသည် အလွန်အသုံးဝင်သည်။ `unsigned` data type များသည် အနုတ်လက္ခဏာမရှိဘဲ အပြုသဘောဆောင်သော ဂဏန်းများကိုသာ ကိုင်တွယ်နိုင်သောကြောင့် Overflow ဖြစ်နိုင်ခြေကို လျော့ချပေးသည်။

## ၅။ Defensive Programming Principles နှင့် Debugging (Defensive Programming and Debugging)

Error Handling နှင့် String စီမံခန့်ခွဲမှုများအပြင် ပရိုဂရမ်ကို ပိုမိုခိုင်မာစေရန်အတွက် Defensive Programming (ကာကွယ်ဆင်ခြင်မှု) မူများကို လက်ခံကျင့်သုံးရန် လိုအပ်သည်။

### Defensive Programming မူများ

Defensive Programming သည် ပရိုဂရမ်ကို အမှားဖြစ်နိုင်ခြေရှိသော အခြေအနေများအတွက် ကြိုတင်ပြင်ဆင်ထားခြင်းနှင့် ထိုအခြေအနေများကို ကိုင်တွယ်ဖြေရှင်းနိုင်ရန် ပရိုဂရမ်ကို ဒီဇိုင်းဆွဲခြင်းကို ဆိုလိုသည်။

ပထမဆုံး မူတစ်ခုမှာ `fail fast` (အမြန်ဆုံး ကျရှုံးခြင်း) မူဖြစ်သည်။ ဆိုလိုသည်မှာ ပရိုဂရမ်သည် အမှားတစ်ခုကို တွေ့ရှိသောအခါ ချက်ချင်း ရပ်တန့်လိုက်ပြီး၊ အမှားကို ဆက်လက်လုပ်ဆောင်ခြင်းမှ ရှောင်ရှားသည်။ အမှားများကို နောက်ပိုင်းတွင် စုဆောင်းပြီးမှ ဖြေရှင်းခြင်းထက် ချက်ချင်းရပ်တန့်ခြင်းက ပရိုဂရမ်၏ အခြေအနေကို ပိုမိုထိန်းချုပ်နိုင်စေသည်။

ဒုတိယမူမှာ `check every return value` (ပြန်ပေးသော တန်ဖိုးတိုင်းကို စစ်ဆေးခြင်း) ဖြစ်သည်။ Function တစ်ခုမှ ပြန်ပေးသော တန်ဖိုးတိုင်းကို လက်ခံရရှိသည့်အခါ၊ ထိုတန်ဖိုးသည် မှန်ကန်သော အခြေအနေဟုတ်မဟုတ် (ဥပမာ- `0` မှန်သည်၊ `-1` မှားသည်) ကို အမြဲတမ်း စစ်ဆေးရမည်။

တတိယမူမှာ `minimize trust` (ယုံကြည်မှုကို အနည်းဆုံးဖြစ်အောင် ထားခြင်း) ဖြစ်သည်။ အခြား module များ၊ အသုံးပြုသူများ သို့မဟုတ် စနစ်မှ ရရှိသော Input များအပေါ် အလွန်အမင်း ယုံကြည်ခြင်းကို ရှောင်ရှားရမည်။ အခြား module များမှ ထွက်လာသော အချက်အလက်များကို လက်ခံရရှိသည့်အခါတိုင်း၊ ၎င်းတို့သည် မှန်ကန်ကြောင်း အမြဲတမ်း စစ်ဆေးရန် လိုအပ်သည်။

### Debugging Technique များ (GDB အခြေခံ)

ပရိုဂရမ်တွင် အမှားများ ရှိနေသည့်အခါ ၎င်းတို့ကို ရှာဖွေပြီး ပြင်ဆင်ရန်အတွက် Debugging (အမှားရှာဖွေခြင်း) သည် မရှိမဖြစ် လိုအပ်သည်။ ဤလုပ်ငန်းစဉ်တွင် `GDB` (GNU Debugger) သည် အလွန်အသုံးဝင်သော Tool တစ်ခုဖြစ်သည်။

`GDB` ကို အခြေခံအသုံးပြုပုံများမှာ အောက်ပါအတိုင်း ဖြစ်သည်။ ပထမဆုံးအနေဖြင့် `breakpoint` ထားခြင်းဖြင့် Code ၏ သတ်မှတ်ထားသော နေရာများတွင် ရပ်တန့်စေသည်။ ၎င်းသည် ပရိုဂရမ်၏ Execution ကို တစ်နေရာရာတွင် ရပ်ဆိုင်းစေသည်။ ထို့နောက် `step` (သို့မဟုတ် `next`) command ကို အသုံးပြု၍ Code ကို တစ်ကြောင်းချင်း အက밟ခြင်းဖြင့် Code တစ်ကြောင်းချင်းစီ၏ Execution ဖြစ်စဉ်ကို လိုက်ကြည့်နိုင်သည်။ ထို့မှသာ အမှားဖြစ်ပေါ်သည့် အချိန်နှင့် နေရာကို တိကျစွာ သိရှိနိုင်သည်။

ထို့အပြင် `print` (သို့မဟုတ် `printf`) command ကို အသုံးပြု၍ ထိုအချိန်တွင် ရှိနေသော `variable` များ၏ တန်ဖိုးများကို စစ်ဆေးနိုင်သည်။ ဤနည်းလမ်းဖြင့် Memory အတွင်းရှိ အချက်အလက်များ မည်သို့ ပြောင်းလဲနေသည်ကို မြင်တွေ့နိုင်သည်။ နောက်ဆုံးတွင် `backtrace` command ကို အသုံးပြု၍ ပရိုဂရမ်သည် မည်သည့် Function များမှ တစ်ဆင့် အဘယ်သို့ Execution လုပ်ဆောင်နေသည်ကို လမ်းကြောင်း (Execution Path) ကို ပြန်လည်စစ်ဆေးနိုင်သည်။ ဤနည်းစနစ်များသည် Memory နှင့် Logic Error များကို ရှာဖွေရာတွင် အလွန်အရေးပါပါသည်။

## ၆။ လက်တွေ့ လေ့ကျင့်ခန်း (Practical Application)

ဤအယူအဆများကို လက်တွေ့ကျကျ အသုံးချနိုင်ရန် အောက်ပါ လက်တွေ့လေ့ကျင့်ခန်းများကို ဆောင်ရွက်ကြည့်ရှုရမည်။

### လက်တွေ့ဥပမာ ၁: Robust User Input Handling Program

ဤလေ့ကျင့်ခန်းတွင် User Input ကို `fgets()` ဖြင့် လက်ခံခြင်း၊ Input Length ကို စစ်ဆေးခြင်းနှင့် Buffer Overflow ကို ကာကွယ်ခြင်းတို့ကို အာရုံစိုက်ရမည်။

```c
#include <stdio.h>
#include <string.h>

int main() {
    char buffer[100]; // Define a buffer to store input
    int result;

    printf("Enter your name (max 99 characters): ");
    
    // Use fgets() for safe input handling
    if (fgets(buffer, sizeof(buffer), stdin) == NULL) {
        printf("Error reading input.\n");
        return 1; // Return an error code
    }

    // Remove the newline character if present
    size_t len = strlen(buffer);
    if (len > 0 && buffer[len - 1] == '\n') {
        buffer[len - 1] = '\0';
    }

    // Input Validation: Check if the input length is reasonable
    if (len >= sizeof(buffer)) {
        printf("Error: Input string is too long.\n");
        return 1; // Indicate error
    }

    printf("You entered: %s\n", buffer);
    
    // Example of returning an error status
    if (result == 0) {
        printf("Input processed successfully.\n");
    } else {
        printf("Input failed processing.\n");
    }

    return 0;
}
```

ဤဥပမာတွင်၊ ကျွန်ုပ်တို့သည် `fgets()` ကို အသုံးပြု၍ Buffer Overflow ကို ကာကွယ်သည်။ `sizeof(buffer)` ကို အသုံးပြုခြင်းဖြင့် Buffer ၏ အမြင့်ဆုံး အရွယ်အစားကို သိရှိပြီး၊ အဲဒီထက် ပိုမိုများများ ရေးထည့်ခြင်းမှ ကာကွယ်သည်။ ထို့အပြင် Input Length ကို စစ်ဆေးခြင်းဖြင့် ပရိုဂရမ်၏ ခိုင်မာမှုကို တိုးတက်စေသည်။

### လက်တွေ့ဥပမာ ၂: GDB ဖြင့် Bug ရှာဖွေခြင်း

ဤဥပမာတွင် အမှားပါသော Code တစ်ခုကို ရေးသားပြီး `GDB` ကို အသုံးပြု၍ Memory/Logic Error များကို ရှာဖွေခြင်းကို ပြသမည်။

```c
// buggy_code.c
#include <stdio.h>

int calculate(int a, int b) {
    int sum = a + b;
    // Intentional bug: forgetting to check for overflow or boundary conditions
    return sum;
}

int main() {
    int x = 2000000000; // A large number, potentially causing overflow issues
    int y = 3000000000;
    
    int result = calculate(x, y);
    
    printf("Result: %d\n", result);
    
    return 0;
}
```

ဤ Code ကို `gcc buggy_code.c -o buggy` ဖြင့် Compile လုပ်ပြီးနောက် `GDB` ကို အသုံးပြု၍ Debug လုပ်မည်။

```bash
gdb ./buggy
(gdb) break main
Breakpoint 1 at 0x...: file buggy_code.c, line 10.
(gdb) run
Starting program: /path/to/buggy 

Breakpoint 1, main () at buggy_code.c:10
10	    int result = calculate(x, y);
```

ယခုအခါ `main` function ၏ ၁၀ ကြိမ်မြောက် နေရာတွင် ရပ်တန့်နေပြီဖြစ်သည်။

```gdb
(gdb) next
calculate (2000000000, 3000000000)
11	    int sum = a + b;
```

`next` command ဖြင့် Function ခေါ်ဆိုမှုကို ဖြတ်ကျော်ပြီး Function အတွင်းသို့ ဝင်ရောက်သည်။

```gdb
(gdb) next
12	    return sum;
```

`return sum;` နေရာသို့ ရောက်ရှိသွားပြီ။ ယခုအခါ `sum` variable ၏ တန်ဖိုးကို စစ်ဆေးကြည့်မည်။

```gdb
(gdb) print sum
$1 = 5000000000  // Note: This might show an incorrect value due to integer overflow behavior
```

ဤနေရာတွင် ကျွန်ုပ်တို့သည် `sum` ၏ တန်ဖိုးကို စစ်ဆေးပြီး၊ ၎င်းသည် `int` data type ၏ ကန့်သတ်ချက်ကို ကျော်လွန်သွားပုံကို မြင်တွေ့ရမည်။ `backtrace` command ကို အသုံးပြု၍ Code ၏ Execution Path ကို ပြန်လည်စစ်ဆေးခြင်းဖြင့် အမှား၏ အရင်းအမြစ်ကို ပိုမိုနက်ရှိုင်းစွာ ရှာဖွေနိုင်မည် ဖြစ်သည်။

## ၇။ နိဂုံးနှင့် အနှစ်ချုပ် (Conclusion and Summary)

ဤအခန်းတစ်လျှောက်လုံးတွင် ကျွန်ုပ်တို့သည် C Programming တွင် Error Handling နှင့် Defensive Programming တို့သည် ပရိုဂရမ်ရေးသားရာတွင် မည်မျှအရေးကြီးကြောင်းကို နားလည်ခဲ့ကြသည်။ C Language တွင် Exception Handling မရှိသောကြောင့်၊ ကျွန်ုပ်တို့သည် Return Value များ၊ System Error Codes များ၊ နှင့် `setjmp/longjmp` ကဲ့သို့သော နည်းလမ်းများကို အသုံးပြု၍ အမှားများကို ကိုင်တွယ်ဖြေရှင်းရသည်။ ထို့အပြင် `assert()` ကို အသုံးပြု၍ Development Time တွင် Assumption များကို စစ်ဆေးခြင်း၊ Input Validation ဖြင့် Buffer Overflow ကဲ့သို့သော လုံခြုံရေးပြဿနာများကို ကာကွယ်ခြင်း၊ String များကို `fgets()` နှင့် `snprintf()` ကဲ့သို့သော ဘေးကင်းသော နည်းလမ်းများဖြင့် စီမံခန့်ခွဲခြင်း၊ နှင့် `fail fast` ကဲ့သို့သော Defensive Programming မူများကို လက်ခံကျင့်သုံးခြင်းတို့သည် ပရိုဂရမ်များ၏ Reliability နှင့် Security ကို သိသိသာသာ တိုးတက်စေသည်။

Debugging လုပ်ရာတွင် `GDB` ကဲ့သို့သော Tools များကို အသုံးပြုခြင်းဖြင့် Code ၏ အတွင်းပိုင်း အခြေအနေများကို စစ်ဆေးနိုင်ခြင်းသည် အမှားများကို ရှာဖွေပြီး ပြင်ဆင်ရာတွင် အလွန်အထောက်အကူပြုသည်။

အချုပ်အားဖြင့်ဆိုရလျှင်၊ C Programming တွင် ထိရောက်သော ပရိုဂရမ်တစ်ခုကို ရေးသားနိုင်ရန်အတွက် Error Handling နှင့် Defensive Programming သည် အခြေခံကျသော ကျွမ်းကျင်မှုများ ဖြစ်သည်။ လာမည့်အခန်းများတွင် ကျွန်ုပ်တို့သည် Data Structures နှင့် File I/O ကဲ့သို့သော အကြောင်းအရာများကို လေ့လာမည်ဖြစ်သော်လည်း၊ ဤအခန်းမှ သင်ယူထားသော နည်းစနစ်များကို အခြေခံအဖြစ် အမြဲတမ်း အသုံးချရန် အရေးကြီးပါသည်။ ပရိုဂရမ်ရေးသားရာတွင် အမှားများကို ကြိုတင်ကာကွယ်ခြင်းနှင့် ထိရောက်စွာ ကိုင်တွယ်ဖြေရှင်းခြင်းသည် ကျွမ်းကျင်သော Software Developer တစ်ဦးဖြစ်လာရန်အတွက် မရှိမဖြစ် လိုအပ်သော အရည်အချင်းများ ဖြစ်ပါသည်။
