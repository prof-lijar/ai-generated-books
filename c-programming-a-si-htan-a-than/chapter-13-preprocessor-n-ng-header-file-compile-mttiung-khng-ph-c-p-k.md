# Chapter 13: Preprocessor နှင့် Header File — Compile မတိုင်ခင် ဖြစ်ပျက်ခြင်း

## ၁။ အခန်းအစနှင့် မိတ်ဆက် (Introduction and Hook)

C Programming တွင် Source Code တစ်ခုကို ရေးသားပြီးနောက် ထိုကုဒ်ကို စက်က အသုံးပြုနိုင်မည့် Machine Code အဖြစ် ပြောင်းလဲပေးသည့် လုပ်ငန်းစဉ်သည် အဆင့်ဆင့် ရှိနေပါသည်။ ဤလုပ်ငန်းစဉ်၏ အစတွင် အရေးကြီးဆုံးနှင့် ပထမဆုံး အဆင့်တစ်ခုမှာ Preprocessor (ကြိုတင်လုပ်ဆောင်သူ) ၏ အခန်းကဏ္ဍ ဖြစ်ပါသည်။ Preprocessor ဆိုသည်မှာ Compiler (ကွန်ပျူတာ ဘာသာပြန်စက်) သည် တိုက်ရိုက်မလုပ်ဘဲ Source Code ကို Compiler အတွက် ပြင်ဆင်ပေးသည့် အခန်းကဏ္ဍကို ဆိုလိုပါသည်။ သင်သည် Source Code ကို ရေးလိုက်သည်နှင့် Compiler က ချက်ချင်း အလုပ်မလုပ်ဘဲ၊ Preprocessor က အရင်ဆုံး လာရောက်စစ်ဆေးပြင်ဆင်ပေးသည်။ ဤအဆင့်တွင် Code ထဲမှ သတ်မှတ်ထားသော အထူးညွှန်ကြားချက်များကို စစ်ဆေးပြီး ထည့်သွင်းခြင်း၊ ဖယ်ရှားခြင်း စသည့် လုပ်ငန်းများကို လုပ်ဆောင်သည်။

Compiler နှင့် Preprocessor တို့၏ ကွာခြားချက်ကို နားလည်ရန် အရေးကြီးပါသည်။ Compiler (ဘာသာပြန်စက်) သည် Preprocessor မှ ထွက်လာသော စစ်ဆေးပြီးသား Code ကို လက်ခံကာ၊ ထို Code ကို Machine Code အဖြစ် ပြောင်းလဲပေးသည့် အဓိက လုပ်ငန်းကို လုပ်ဆောင်သည်။ ဥပမာအားဖြင့်၊ သင်သည် `#include` instruction ကို ရေးလိုက်သည့်အခါ၊ Preprocessor သည် ထို instruction ကို ဖတ်ပြီး၊ သတ်မှတ်ထားသော Header File တစ်ခုလုံး၏ အကြောင်းအရာကို ထိုနေရာတွင် ကူးထည့်ပေးသည်။ ထို့နောက်မှ Compiler သည် ထိုပြီးပြည့်စုံသော Code ကို Machine Code အဖြစ် ပြောင်းလဲပေးသည်။ ဤကွာခြားချက်ကို နားလည်ခြင်းက C Programming ၏ အခြေခံ လုပ်ဆောင်ပုံကို နားလည်ရန် အထောက်အကူပြုသည်။

## ၂။ Preprocessor နှင့် Header File များ (The Core of Preprocessing)

Preprocessor သည် Source Code ကို Compiler ထံသို့ ပေးပို့မည့် အဆင့်တွင် အဓိက အခန်းကဏ္ဍမှ ပါဝင်သည်။ ဤအဆင့်တွင် အဓိကအားဖြင့် `#include` directive နှင့် `#define` macro များအပြင်၊ Code ၏ တည်ငြိမ်မှုကို ထိန်းသိမ်းရန်အတွက် Conditional Compilation နှင့် Include Guards များက အရေးကြီးပါသည်။

### `#include` Directive နှင့် Header File များ (Header Files)

`#include` directive သည် Preprocessor အတွက် အလွန်အရေးကြီးသော အမိန့်တစ်ခု ဖြစ်သည်။ ၎င်းသည် သတ်မှတ်ထားသော ဖိုင်တစ်ခုလုံး၏ အကြောင်းအရာများကို လက်ရှိ Source File ထဲသို့ ထည့်သွင်းရန် အသုံးပြုသည်။ Header File များသည် ဤလုပ်ငန်းစဉ်တွင် အဓိက အခန်းကဏ္ဍမှ ပါဝင်သည်။

`#include` ကို အသုံးပြုရာတွင် Header File နှစ်မျိုးကို ခွဲခြားသိရှိရန် လိုအပ်သည်။ ပထမတစ်ခုမှာ `< >` ကွင်းစကွင်းအတွင်းတွင် ရေးသားသော Header များဖြစ်ပြီး ၎င်းတို့ကို Standard Library (စံစာအုပ်များ) တွင် ပါဝင်သော Header များဟု သတ်မှတ်သည်။ ဥပမာအားဖြင့်၊ `<stdio.h>` သည် Standard Input/Output functions များအတွက် လိုအပ်သော အချက်အလက်များ ပါဝင်သည်။ ဤ Header များသည် Compiler နှင့် အတူတကွ အလုပ်လုပ်ပြီး၊ ထို Library များမှ Function Prototypes နှင့် Type Definitions များကို ရယူသည်။

ဒုတိယတစ်ခုမှာ `"` ကွင်းစကွင်းအတွင်းတွင် ရေးသားသော Header များဖြစ်ပြီး ၎င်းတို့ကို User-defined Header များဟု ခေါ်သည်။ ဥပမာအားဖြင့်၊ `"myheader.h"` သည် သင်ကိုယ်တိုင် ဖန်တီးထားသော Header File ဖြစ်သည်။ ဤ Header File များသည် သင့်ပရောဂျက်၏ သီးခြားလိုအပ်ချက်များနှင့် သက်ဆိုင်သော အချက်အလက်များ၊ Function Prototypes များ သို့မဟုတ် Macro Definitions များကို ထည့်သွင်းရန် အသုံးပြုသည်။

Header File များ၏ အခန်းကဏ္ဍမှာ အလွန်ကျယ်ပြန့်ပါသည်။ ၎င်းတို့သည် Function Prototypes (လုပ်ဆောင်ချက်များ၏ အမည်၊ လက်ခံမည့် Argument များနှင့် ပြန်ပေးမည့် အဖြေများ) နှင့် Macro Definitions များကို Source File များအကြားတွင် ကြိုတင်သတ်မှတ်ပေးခြင်းဖြင့် Code ရေးသားရာတွင် အမှားနည်းစေပြီး ပိုမိုရှင်းလင်းစေသည်။

### Macro Definitions နှင့် `#define` (မက်ခရိုများ)

`#define` directive သည် Preprocessor ၏ အင်အားအကြီးဆုံး လုပ်ဆောင်ချက်များထဲမှ တစ်ခုဖြစ်သည်။ ၎င်းသည် Text Substitution (စာသားအစားထိုးခြင်း) ကို အခြေခံသည်။

`#define` ကို အသုံးပြု၍ `Constant` များ ဖန်တီးခြင်းသည် အလွန်အသုံးဝင်သည်။ ပုံမှန်အားဖြင့် `const` keyword ကိုသုံး၍ တန်ဖိုးများကို သတ်မှတ်သော်လည်း၊ `#define` ကို အသုံးပြုခြင်းဖြင့် ပိုမိုကျယ်ပြန့်သော အကျိုးကျေးဇူးများ ရရှိသည်။ ဥပမာအားဖြင့်၊ သင်သည် ပရောဂျက်တစ်ခုတွင် အသုံးပြုမည့် အဓိက တန်ဖိုး (ဥပမာ: $\pi$ တန်ဖိုး၊ အမြင့်ဆုံး အကန့်အသတ်) များကို သတ်မှတ်လိုပါက `#define` ကို အသုံးပြုနိုင်သည်။ ၎င်းသည် ကုတ်ကို ဖတ်ရလွယ်ကူစေသည်။

ထို့အပြင်၊ `Function-like Macro` များ ဖန်တီးခြင်းသည် Code ကို အစားထိုးခြင်းနှင့် ဆင်တူသည်။ ၎င်းသည် ရှုပ်ထွေးသော Code Block များကို ရိုးရှင်းသော အမည်တစ်ခုဖြင့် အစားထိုးပေးသည်။ ဥပမာအားဖြင့်၊ ရှည်လျားသော `if-else` statement တစ်ခုကို Macro တစ်ခုအဖြစ် သတ်မှတ်ခြင်းဖြင့် Code ကို အတိုချုပ်နိုင်သည်။

Macro များ၏ အကျိုးကျေးဇူးမှာ Code ကို ထပ်ခါထပ်ခါ ရေးသားရခြင်းကို ရှောင်ရှားပြီး ပိုမိုထိရောက်စေသည်။ သို့သော် Macro များ၏ အန္တရာယ်ကိုလည်း သတိထားရမည်။ Macro များတွင် Side Effects (အခြား Variable များအပေါ် သက်ရောက်မှုများ) များ ဖြစ်ပေါ်လာနိုင်ပြီး၊ အထူးသဖြင့် Argument များပေးပို့သောအခါ အမှားများ ဖြစ်ပေါ်စေနိုင်သည်။ ထို့ကြောင့် Macro များကို သတိရှိစွာ အသုံးပြုရန် လိုအပ်သည်။

### Conditional Compilation နှင့် Include Guards

Source Code ၏ တည်ငြိမ်မှုကို ထိန်းသိမ်းရန်အတွက် Conditional Compilation နှင့် Include Guards များသည် မရှိမဖြစ်လိုအပ်သည်။

`#ifdef`, `#ifndef`, နှင့် `#endif` တို့သည် Source Code ၏ အချို့အပိုင်းများကို သတ်မှတ်ထားသော အခြေအနေများအပေါ် မူတည်၍ ထည့်သွင်းခြင်း သို့မဟုတ် ဖယ်ရှားခြင်းတို့ကို လုပ်ဆောင်သည်။ ဤအရာများသည် Compiler ကို ပေးမည့် Code ၏ ပုံစံကို ပြောင်းလဲစေသည်။ ဥပမာအားဖြင့်၊ `#ifdef` သည် အချို့သော မဲလ်တီပရိုဂရမ်များတွင်သာ လိုအပ်သော Code ကို ထည့်သွင်းရန် အသုံးပြုသည်။

အထူးအရေးကြီးသော အချက်မှာ Include Guards များဖြစ်သည်။ Header File များစွာကို `#include` လုပ်သောအခါ၊ တစ်ခုနှင့်တစ်ခု ထပ်နေခြင်း (Multiple Inclusion) ပြဿနာသည် အလွယ်တကူ ဖြစ်ပေါ်တတ်သည်။ ဤပြဿနာကို ရှောင်ရှားရန်အတွက် Include Guards ကို အသုံးပြုသည်။ Include Guards များမှာ `#ifndef`, `#define`, နှင့် `#endif` ကို အသုံးပြု၍ ဖန်တီးသည်။ ၎င်းတို့သည် File တစ်ခုကို အကြိမ်ကြိမ် ထည့်သွင်းခြင်းကို ကာကွယ်ပေးသည်။

Include Guard များ မရှိပါက ဖြစ်ပေါ်လာမည့် ပြဿနာကို ဥပမာပေးရမည်ဆိုလျှင်၊ သင်သည် Header File တစ်ခုတွင် Function Prototypes များကို ထည့်သွင်းထားပြီး၊ ထို Header File ကို Source File နှစ်ခုလုံးတွင် `#include` လုပ်လိုက်ပါက၊ ထို Function Prototypes များသည် ထပ်ခါထပ်ခါ ထည့်သွင်းခံရမည်။ ၎င်းသည် Compiler ကို မှားယွင်းသော အချက်အလက်များ ပေးပို့ပြီး၊ အဆုံးတွင် Segmentation Fault သို့မဟုတ် အခြား Runtime Error များ ဖြစ်ပေါ်စေနိုင်သည်။ ထို့ကြောင့် Include Guards များသည် Source Code ၏ တည်ငြိမ်မှုကို အာမခံပေးသည်။

## ၃။ ကိုယ်ပိုင် Header File များ ဖန်တီးခြင်း (Creating Custom Header Files)

အဆင့် (၂) တွင် Header File များ၏ အခန်းကဏ္ဍကို လေ့လာပြီးနောက်၊ လက်တွေ့တွင် ကိုယ်ပိုင် Header File များ ဖန်တီး၍ Multi-file Project များအတွက် အသုံးပြုပုံကို လေ့လာရမည်ဖြစ်သည်။

### Declaration နှင့် Definition များကို ခွဲခြားခြင်း

Header File များတွင် အချက်အလက်များကို မည်သို့ စီစဉ်ထားသည်ကို နားလည်ရန် Declaration (ကြေညာချက်) နှင့် Definition (အဓိပ္ပာယ်ဖွင့်ဆိုချက်) တို့ကို ခွဲခြားသိရှိရန် လိုအပ်သည်။

Header File များတွင် `struct`, `enum`, နှင့် `typedef` ကဲ့သို့သော Data Structures များကို ထည့်သွင်းရာတွင် ဤခွဲခြားမှုသည် အရေးကြီးသည်။

Declaration (ကြေညာချက်) ဆိုသည်မှာ အရာဝတ္ထုတစ်ခု၏ အမည်နှင့် အခြေခံအချက်အလက်များကို ဖော်ပြခြင်း ဖြစ်သည်။ ဥပမာအားဖြင့်၊ `struct` တစ်ခုကို ကြေညာခြင်းသည် ထိုအရာဝတ္ထုအမည်ကို သတ်မှတ်ခြင်းသာ ဖြစ်သည်။

Definition (အဓိပ္ပာယ်ဖွင့်ဆိုချက်) ဆိုသည်မှာ ထိုအရာဝတ္ထု၏ အတိအကျ အဓိပ္ပာယ်နှင့် Memory Allocation ကို သတ်မှတ်ခြင်း ဖြစ်သည်။

Header File များတွင် `struct` များသည် Data Structure များ၏ ပုံစံကို ဖော်ပြသည်။ ၎င်းတို့ကို Header File တွင် ထည့်သွင်းခြင်းသည် ထို Data Structure များ၏ ပုံစံကို အခြား Source File များတွင် အသုံးပြုနိုင်ရန် ခွင့်ပြုသည်။

`extern` keyword (အပြင်မှ) သည် အချက်အလက်ကို တခြားနေရာမှ ရယူခြင်းကို ညွှန်ပြသည်။ ၎င်းသည် အချက်အလက်ကို Header File တွင် ကြေညာသော်လည်း၊ ထိုအချက်အလက်၏ တည်ရှိရာနေရာ (Definition) ကို နောက်ပိုင်းတွင် အခြား Source File တစ်ခုတွင် သတ်မှတ်မည်ကို ဆိုလိုသည်။ ၎င်းသည် Module များအကြား အချက်အလက်များအား အပြန်အလှန် ဆက်သွယ်နိုင်ရန် အထောက်အကူပြုသည်။

### Multi-file Project များအတွက် ပြင်ဆင်ခြင်း

Source File များစွာဖြင့် Program ရေးခြင်းသည် ပရောဂျက်ကြီးများအတွက် မရှိမဖြစ်လိုအပ်သည်။ တစ်ခုတည်းသော Source File ဖြင့် ရေးသားခြင်းသည် ရိုးရှင်းသော်လည်း ရှုပ်ထွေးသော ပရောဂျက်များတွင် စီမံခန့်ခွဲရန် ခက်ခဲသည်။ Source File များစွာဖြင့် ရေးသားခြင်း၏ အကျိုးမှာ Code ကို အပိုင်းလိုက် ခွဲ၍ စီမံခန့်ခွဲနိုင်ခြင်း၊ ပရောဂျက်၏ အစိတ်အပိုင်းအလိုက် ပြုပြင်နိုင်ခြင်းနှင့် ပိုမိုရှင်းလင်းသော Code ကို ရရှိနိုင်ခြင်းတို့ ဖြစ်သည်။

ဤ Multi-file Project များတွင် Header File နှင့် Source File များ၏ သဘောတရားကို နားလည်ရန် အရေးကြီးသည်။ Header File များသည် ပရောဂျက်၏ Interface (မျက်နှာစာ) အဖြစ် ဆောင်ရွက်ပြီး၊ Source File များသည် အမှန်တကယ် Logic (အလုပ်လုပ်ပုံ) များကို ထည့်သွင်းသည်။

`extern` keyword ကို အသုံးပြုခြင်းသည် Module များအကြား ဆက်သွယ်ပုံကို အဓိကကျသည်။ Header File များတွင် `extern` ကို အသုံးပြု၍ အချက်အလက်များကို ကြေညာပြီး၊ Source File များတွင် ထိုအချက်အလက်များကို အသုံးပြုရန်အတွက် `extern` ကို အသုံးပြုသည်။ ဤနည်းလမ်းဖြင့်၊ အချက်အလက် တစ်ခုတည်းကို အများအပြားသော File များတွင် အသုံးပြုနိုင်ပြီး၊ Memory ကို ထိရောက်စွာ အသုံးပြုနိုင်သည်။ Header File သည် အချက်အလက်များ၏ ကြေညာချက်ကိုပေးပြီး၊ Source File များသည် ထိုအချက်အလက်များကို အမှန်တကယ် ဖန်တီးသည်။

## ၄။ Compilation Process နှင့် Automation (Making it Efficient)

Preprocessor နှင့် Header File များသည် Code ကို ပြင်ဆင်ပေးသော်လည်း၊ ထို Code ကို Machine Code အဖြစ် ပြောင်းလဲပေးသည့် နောက်ဆုံးအဆင့်မှာ Compiler နှင့် Linker တို့၏ အခန်းကဏ္ဍများဖြစ်သည်။ ဤအဆင့်များကို ပိုမိုထိရောက်စွာ စီမံခန့်ခွဲရန်အတွက် Compilation Process ကို အလိုအလျောက် (Automation) လုပ်ဆောင်ခြင်းသည် အလွန်အရေးကြီးလာသည်။

### Source File များစွာကို စီမံခန့်ခွဲခြင်း

Multiple `source file` များဖြင့် Project တည်ဆောက်ပုံသည် ပရောဂျက်ကြီးများအတွက် မရှိမဖြစ်လိုအပ်သည်။ ပရောဂျက်တစ်ခုသည် အစိတ်အပိုင်းအမျိုးမျိုးဖြင့် စုစည်းထားသောအခါ၊ Source File များသည် ထိုအစိတ်အပိုင်းများကို သီးခြားစီ ခွဲခြားထားခြင်းဖြစ်သည်။

Compiler သည် Source File အားလုံးကို မည်သို့ Process လုပ်ပုံကို နားလည်ရန် လိုအပ်သည်။ Compiler သည် Source File တစ်ခုချင်းစီကို သီးခြားစီ Process လုပ်ပြီး၊ ထို Process များမှ ရရှိသော Intermediate Code များကို နောက်ဆုံးတွင် Linker သို့ ပေးပို့သည်။ ဤလုပ်ငန်းစဉ်တွင် Source File များသည် တစ်ခုနှင့်တစ်ခု မည်သို့ ဆက်စပ်နေသည်ကို Compiler သည် သိရှိရန် လိုအပ်သည်။

Source File တစ်ခုစီမှ ထွက်ပေါ်လာသော အပိုင်းအစများကို `object file` များဟု ခေါ်သည်။ ဤ `object file` များသည် နောက်ဆုံးတွင် Linker ၏ အခန်းကဏ္ဍမှ ပေါင်းစပ်ပြီး `executable` (အသုံးချနိုင်သော ဖိုင်) အဖြစ် ပြောင်းလဲသွားသည်။ ဤ `object file` နှင့် `executable` တို့အကြား ဆက်စပ်မှုကို နားလည်ခြင်းသည် C Programming ၏ လုပ်ငန်းစဉ်ကို နားလည်ရန် အခြေခံဖြစ်သည်။

### Makefile အခြေခံများ (Introduction to Makefile)

Compilation Process ကို **Automate** လုပ်ခြင်းသည် ပရောဂျက်ကြီးများအတွက် အလွန်အရေးကြီးသည်။ လက်ဖြင့် Compile လုပ်ခြင်းသည် အချိန်ကုန်ပြီး အမှားများ ဖြစ်ပေါ်လွယ်သည်။ ထို့ကြောင့် `Makefile` ဆိုသည်မှာ Compilation Process ကို အလိုအလျောက် လုပ်ဆောင်ပေးမည့် စာသားဖိုင်ဖြစ်သည်။

`Makefile` ၏ အခြေခံဖွဲ့စည်းပုံမှာ Rules (စည်းမျဉ်းများ) နှင့် Targets (ပန်းတိုင်များ) များဖြင့် ဖွဲ့စည်းထားသည်။ Rule များသည် ဘယ်လိုအခြေအနေများတွင် ဘယ်လိုလုပ်ဆောင်ရမည်ကို ဖော်ပြသည်။ Targets များသည် လုပ်ဆောင်ရမည့် အဆုံးသတ် ရည်မှန်းချက်များ ဖြစ်သည်။

Compilation, Linking နှင့် Dependency များအပေါ် အခြေခံ၍ `Makefile` ရေးဆွဲပုံကို အကျဉ်းမျှ ဖော်ပြရမည်။ `Makefile` တွင် Compilation လုပ်ရန်အတွက် Source File များ၊ Header File များနှင့် Linker များအကြောင်းကို ဖော်ပြရမည်။ Dependency ဆိုသည်မှာ အခြားအရာတစ်ခု၏ အပေါ်တွင် မှီခိုနေသော အရာများ ဖြစ်သည်။ ဥပမာအားဖြင့်၊ Source File တစ်ခုကို ပြောင်းလဲပါက၊ ထို Source File နှင့် ထို Source File မှ မှီခိုနေသော Object File များအား ပြန်လည် Compile လုပ်ရန် လိုအပ်သည်။ `Makefile` သည် ဤ Dependency များကို အလိုအလျောက် စီမံခန့်ခွဲပေးခြင်းဖြင့် Compilation Process ကို အလွန်ထိရောက်စေသည်။

## ၅။ လက်တွေ့ လေ့ကျင့်ခန်းနှင့် အကျဉ်းချုပ် (Practical Application and Summary)

အထက်ပါ အကြောင်းအရာများကို လက်တွေ့ကျကျ လေ့ကျင့်ပြီး နားလည်ရန်အတွက် လက်တွေ့ Project တစ်ခုကို ဖန်တီးခြင်းသည် အကောင်းဆုံး နည်းလမ်းဖြစ်သည်။

### လက်တွေ့ Project ဖန်တီးခြင်း (Hands-on Exercise)

Header File နှင့် Multiple Source File များဖြင့် အသေးစား Project တစ်ခုကို အတူတကွ ဖန်တီးခြင်းသည် အကောင်းဆုံး လေ့ကျင့်ခန်းဖြစ်သည်။ ဥပမာအားဖြင့်၊ သုံးစွဲသူ၏ အချက်အလက်များကို သိမ်းဆည်းပြီး တွက်ချက်မှုများ ပြုလုပ်သော အသေးစား ပရောဂျက်တစ်ခုကို ဖန်တီးနိုင်သည်။ ဤပရောဂျက်တွင် အချက်အလက်များ၏ ပုံစံကို Header File တွင် သတ်မှတ်ပြီး၊ ထိုအချက်အလက်များကို အခြား Source File များမှ အသုံးပြုမည်။ ဤလုပ်ငန်းစဉ်တွင် Preprocessor မှ မှားယွင်းမှုများ ရှိမရှိ စစ်ဆေးခြင်းနှင့် Debugging လုပ်ခြင်းသည် အရေးကြီးသည်။

### Chapter Summary

ဤအခန်းတွင် Preprocessor နှင့် Header File များ၏ အခန်းကဏ္ဍများကို ပြန်လည်အကျဉ်းချုပ်ရမည်။ Preprocessor သည် Compiler မလုပ်ခင် Source Code ကို ပြင်ဆင်ပေးသည့် အရေးပါသော အဆင့်ဖြစ်ပြီး၊ `#include` နှင့် `#define` များက Code ကို တိုးချဲ့ပေးသည်။ Include Guards များက Code ၏ တည်ငြိမ်မှုကို အာမခံပေးသည်။ ကိုယ်ပိုင် Header File များ ဖန်တီးခြင်းနှင့် `extern` keyword ကို အသုံးပြုခြင်းဖြင့် Multi-file Project များအတွက် အခြေခံအုတ်မြစ်ချပေးသည်။ နောက်ဆုံးတွင်၊ Source File များစွာကို စီမံခန့်ခွဲရန်အတွက် `Makefile` ကဲ့သို့သော Automation နည်းစနစ်များကို အသုံးပြုခြင်းသည် Compilation Process ကို အလွန်ထိရောက်စေသည်။

Compiler နှင့် Linker တို့သည် Preprocessor မှ ထွက်လာသော Code ကို လက်ခံပြီး၊ နောက်ဆုံးတွင် Machine Code အဖြစ် ပြောင်းလဲကာ Executable ဖိုင်အဖြစ် ထုတ်ပေးသည်။ Preprocessor သည် Code ကို ပြင်ဆင်ခြင်း၊ Compiler သည် ဘာသာပြန်ခြင်း၊ Linker သည် အစိတ်အပိုင်းများကို ပေါင်းစပ်ခြင်းတို့ဖြင့် အတူတကွ လုပ်ဆောင်ကာ ကျွန်ုပ်တို့၏ C Program ကို လက်တွေ့ကွန်ပျူတာတွင် အလုပ်လုပ်စေသည်။ ဤအခြေခံအုတ်မြစ်များသည် Pointer၊ Memory Management နှင့် ပိုမိုရှုပ်ထွေးသော Data Structures များ လေ့လာရန်အတွက် မရှိမဖြစ်လိုအပ်သော အခြေခံများ ဖြစ်သည်။
